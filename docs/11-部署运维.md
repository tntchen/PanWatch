# 11 · 部署、运维与工程化

> 面向二次开发者：本文覆盖 PanWatch 的构建、发布、开发环境、测试体系与 CI/CD。
> 范围文件：Dockerfile、build.sh、Makefile、pyproject.toml、requirements.txt、scripts/、.github/、tests/conftest.py（+抽查测试）、CONTRIBUTING.md、CLAUDE.md、VERSION、.dockerignore、.gitignore、.env.example。

## 一句话总结

PanWatch 是一个"单容器自托管"项目：**前端 pnpm 构建产物打进多阶段 Docker 镜像、后端 `python server.py` 单进程托管静态文件 + API + APScheduler**，本地开发用 Makefile 一键起前后端，测试用 pytest（默认屏蔽通知），发布走 GitHub Actions 打 tag 构建多架构镜像并推 Docker Hub。

## 部署形态与运行机制

### Docker 单容器（生产主路径）

- 两阶段：Stage1 `node:20-alpine` 跑 `pnpm install --frozen-lockfile && pnpm build`；Stage2 `python:3.11-slim` 装系统依赖 + pip 依赖 + 拷贝 `src/`、`server.py`、`prompts/`、`packages/`，前端产物落在 `/app/static/`。
- 系统依赖分三类：`tzdata`（zoneinfo 需要）、`fonts-noto-cjk`（K线截图/PDF 中文）、一大批 Chromium 系统库（Playwright）、`git`（pip 需要拉 tradingagents 的 git 直链）。
- 环境变量：`DATA_DIR=/app/data`、`DOCKER=1`、`TZ=Asia/Shanghai`（可 `-e TZ=` 覆盖）、`PYTHONUNBUFFERED=1`。
- **Playwright 浏览器不打进镜像**：`server.py` 启动时检测（约 212-259 行），未设 `PLAYWRIGHT_BROWSERS_PATH` 时默认装到 data 目录下并设该环境变量，执行 `playwright install chromium`；可用 `PLAYWRIGHT_SKIP_BROWSER_INSTALL=1` 跳过。这样镜像更小且浏览器跨版本持久化在卷里。
- 数据持久化：`/app/data`（SQLite `panwatch.db`、JSON、Playwright 浏览器、企业 CA 证书 `ca-bundle.pem`），官方运行命令 `docker run -d -p 8000:8000 -v panwatch_data:/app/data ...`。
- 健康检查：`HEALTHCHECK` 每 30s 请求 `/api/health`（路由在 `src/web/app.py`）。
- 静态文件：`server.py` 末尾（约 1542 行）检测 `./static` 存在则用 `StaticFiles`/`FileResponse` 托管，前端 SPA 与后端 API 同端口 8000。

### 版本流

tag（格式 `0.2.3`，无 v 前缀）→ release.yml → 镜像 tag 同时打 `{version}` 和 `latest`（可关），VERSION 文件在镜像内由 build-arg 写入。`build.sh` 本地路径只构建 amd64；CI 默认 `linux/amd64,linux/arm64`。

## CI/CD（.github/workflows/release.yml）

触发：push 符合 `*.*.*` 的 tag，或 `workflow_dispatch`（可指定版本/是否推 latest/平台）。

步骤顺序（任一失败即中断发布）：

1. Python 3.11 + 安装 WeasyPrint 系统库（pango/cairo/noto-cjk）→ `pip install -r requirements.txt`
2. `pytest tests/ -x -q`（主测试套件）
3. `pytest packages/marketdata/tests -q`（本地行情子包独立测试）
4. QEMU + Buildx 登录 Docker Hub（secrets: `DOCKERHUB_USERNAME`/`DOCKERHUB_TOKEN`），多架构 build & push，`VERSION` 作 build-arg，缓存走 `type=gha`
5. 仅 tag push 时：生成 release notes（与上一 tag 的 `git log --no-merges`），`softprops/action-gh-release` 建 GitHub Release，并可选发 Telegram 通知（secrets: `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID`，`continue-on-error`）

镜像名硬编码为 `sunxiao0721/panwatch`（build.sh、release.yml、`src/core/update_checker.py` 三处）。

## 测试体系

- 配置在 `pyproject.toml`：`testpaths=["tests"]`，约 67 个测试文件，命名 `test_*.py`，**unittest 和 pytest 风格混用**。
- `conftest.py` 关键机制：
  - `pytest_itemcollected` hook：用测试函数中文 docstring 首行替换 nodeid，`pytest -v` 输出为中文说明 —— **新测试必须写中文 docstring**。
  - autouse fixture 默认把 `NotifierManager.notify / notify_with_result` 替换为 AsyncMock，**防止单测误发真实通知**；`--notify` 选项恢复真实发送。
  - autouse fixture mock `stock_link.get_platform`（避免测试读 DB）、清空 K线/资金流内存缓存（防用例间污染）。
  - session 级 fixture 幂等 `Base.metadata.create_all(engine)`，保证 CI 全新环境 `data/panwatch.db` 有表。
- 子包独立测试：`packages/marketdata/tests/`（24 个文件，覆盖 vendor/cache/client/symbol 等），CI 单独跑。
- 抽查观感：测试偏单元级，重边界（cron 星期换算 POSIX→APScheduler、时区 UTC 换算、通知策略 HH:MM 解析、PDF 导出中文入文本层）；`test_pdf_export.py` 会探测 WeasyPrint 是否可渲染，不可用则回退 xhtml2pdf 路径。

## 二次开发扩展点与注意事项

### 新增 Agent / 数据源（CONTRIBUTING.md 定义的官方路径）

1. Agent：`src/agents/` 新文件继承 `BaseAgent`，实现 `collect / build_prompt`（可选 `should_notify`），配 `prompts/xxx.txt`，然后在 `server.py` 两处注册——`AGENT_REGISTRY` 字典 + `seed_agents()` 配置（含 cron、`execution_mode: batch|single`）。
2. 数据源：`src/collectors/` 新 collector（有 `source` 标识，从 DB `DataSource.config` 读配置），在 `server.py` 的 `seed_data_sources()` 注册（`type` ∈ quote/kline/news/capital_flow/chart，`priority` 越小越优先）。

### 容易踩坑的地方

- **requirements.txt 的 git 直链**：`tradingagents @ git+...@v0.3.0` 意味着 pip 安装需要 git 和网络可达 GitHub；首次安装拉 ~115 个包、2-5 分钟。升级它要验证 `src/agents/tradingagents*` 适配层的 monkeypatch 目标符号（注释里点名 route_to_vendor/load_ohlcv 等）。
- **`-e ./packages/marketdata`**：本地 editable 依赖，Dockerfile 必须先 `COPY packages/` 再 `pip install`；改包结构时注意两个 Docker/CI 步骤顺序。该子包有自己的 pyproject 和 tests，变更需同时保证 `pytest packages/marketdata/tests` 通过。
- **端口约定**：后端 8000（Docker/本地统一，不能随意改，Dockerfile 注释明确"避免影响存量用户升级"）、前端 dev 5183（特意与 5173 错开）。
- **WeasyPrint 系统库**：本地跑 PDF 相关测试需要 pango/cairo + 中文字体，否则自动走 xhtml2pdf 回退；CI 的"Install system libs"步骤照抄即可。
- **pre_release_check.sh 依赖 `rg`**（ripgrep）和运行中的实例（smoke 模式），在 macOS/精简环境先确认工具存在。
- **cron 时区/星期换算**：POSIX cron 周日为 0/7、APScheduler 周一为 0，`schedule_parser` 做了归一化（有专门测试），改调度逻辑不要绕过它。
- **测试默认屏蔽通知**：写涉及通知的测试时，断言应对 mock 生效；不要依赖真实渠道。`--notify` 仅用于本地集成验证。
- **`.dockerignore` 排除所有 *.md（除 README）**：不要把运行时需要读取的文件命名成 .md 放根目录（prompts/ 是 .txt 不受影响）。
- **build.sh 只出 amd64**：Apple Silicon 本地构建的镜像在 arm 机器上跑不了原生；多架构发布只有 CI 路径。
- **镜像名硬编码三处**：fork 项目要同步改 `build.sh`、`release.yml`、`src/core/update_checker.py` 的默认 repo，否则更新检查指向原作者镜像。
- **VERSION 文件**：运行时按 cwd 读取，`server.py` 必须从仓库根（或 `/app`）启动，否则 `get_app_version()` 读不到会落到默认值（代码内还有第三处兜底，未细读）。

### 未覆盖/不确定项

- 无 `docker-compose.yml`、无 lint/format 工具配置（ruff/black/eslint 均未在依赖与 CI 中出现，CONTRIBUTING 提到"通过 lint 检查"但无对应工具链）。
- `pyproject.toml` 无打包元数据，后端不可 pip 安装，只能脚本方式运行。
- 无 staging/灰度机制，release.yml 直接推 Docker Hub `latest`。
