# 06 · 通知与调度

> 范围：`src/core/notifier.py`、`notify_policy.py`、`notify_dedupe.py`、`scheduler_registry.py`、`context_scheduler.py`、`paper_trading_scheduler.py`、`price_alert_scheduler.py`、`schedule_parser.py`、`agent_runs.py`、`analysis_history.py`、`paper_trading_notifier.py`。范围外文件（`scheduler.py`、`agents/base.py`、`server.py` 等）仅做浅读以确认调用关系。

## 一句话总结

本模块负责**多渠道消息推送**（Apprise + 自定义实现，含静默时段、重试、幂等去重）和**五套 APScheduler 调度**（Agent 定时任务、价格提醒、模拟盘、上下文维护，另加调度器健康注册表），并提供 Agent 运行记录与分析历史的落库查询。

## 关键文件清单

### 通知侧

| 文件 | 作用 |
|---|---|
| `notifier.py` | 通知核心。`CHANNEL_TYPES` 定义 9 种渠道及表单字段；`build_apprise_url()` 把配置转成 Apprise URL；`NotifierManager` 统一发送，内部区分 Apprise 渠道（telegram/bark/dingtalk/lark/discord/pushover）与自定义渠道（wecom/serverchan/pushplus + 带代理的 telegram）。 |
| `notify_policy.py` | 冻结 dataclass `NotifyPolicy`：时区、静默时段（`HH:MM-HH:MM`，支持跨午夜）、重试次数/退避、按 Agent 的去重 TTL 覆盖（JSON 字符串解析）。 |
| `notify_dedupe.py` | 通知幂等。`build_notify_dedupe_key()` 对 `agent|title|content(前1200字符)` 做 sha1；`check_and_mark_notify()` 复用 `notify_throttle` 表做 TTL 去重，scope 存在 `stock_symbol` 字段里（形如 `__notify__:<sha1>`）。 |
| `paper_trading_notifier.py` | 模拟盘专属通知：建仓/平仓实时推送、盘前计划、日终摘要、测试通知。自建一套 `pt_notify_*` 配置与渠道选择逻辑，不复用 server.py 的 `_build_notifier`。 |

### 调度侧

| 文件 | 作用 |
|---|---|
| `schedule_parser.py` | 调度表达式解析。5 段 cron（POSIX 语义 day_of_week：0/7=周日）→ APScheduler CronTrigger（0=周一），`normalize_cron_day_of_week_field()` 做数值归一化；`interval:30s/3m/1h` → IntervalTrigger；另提供 `preview_schedule()`（预览未来 N 次触发）和 `count_runs_within()`（统计区间内触发次数）。 |
| `scheduler_registry.py` | 进程内全局注册表（21 行），各调度器 `start()` 时 `register(name, scheduler)`，供 `selfcheck.probe_scheduler()` 读取健康状态；CLI 等无调度进程注册表为空，自检优雅跳过。 |
| `price_alert_scheduler.py` | 价格提醒：默认 60s 间隔调用 `price_alert_engine.ENGINE.scan_once()`；`trigger_once()` 支持 dry_run 和指定规则、绕过交易时段。 |
| `paper_trading_scheduler.py` | 模拟盘：60s 间隔调用 `paper_trading_engine.ENGINE.scan_once()`（全市场休市时跳过）；另挂 cron 盘前计划 09:00、日终摘要 15:30（调用 `paper_trading_notifier`）。 |
| `context_scheduler.py` | 上下文维护：每 6h 后验评估（预测/候选/策略信号 outcome + 策略调权 + 因子自校准）；每天 04:15 清理过期数据；01:15/05:30/14:00 UTC（北京 09:15/13:30/22:00）刷新机会池；启动 15s 后补跑一次评估。 |
| `agent_runs.py` | `record_agent_run()`：把一次 Agent 运行（状态/结果/错误/耗时/trace_id/触发源/通知是否尝试与成功/context 字符数/模型标识）写入 `agent_runs` 表，字段均截断（result/error 2000 字符）。 |
| `analysis_history.py` | `save_analysis()` 等：分析结果按 `(agent_name, stock_symbol, analysis_date)` 唯一，**当天可覆盖、历史不可改**；`get_latest_ta_verdict()` 抽取 TradingAgents 最近结论的紧凑 dict（rating/action_label/one_liner/date/age_days），供盘前盘后做先验，全程 fail-soft。 |

## 核心机制与数据流

### 通知发送链（Agent 路径）

1. `server.py::_build_notifier(channels)`：从 AppSettings 表（回落到 `src/config.py` 的 env 默认值）读 `notify_quiet_hours` / `notify_retry_attempts` / `notify_retry_backoff_seconds` / `notify_dedupe_ttl_overrides`，构造 `NotifyPolicy` 注入 `NotifierManager`。
2. `agents/base.py::run()` 中依次过四道闸：`suppress_notify` → `should_notify()`（子类可重写）→ 静默时段（`policy.is_quiet_now()`）→ 幂等去重（`check_and_mark_notify`，先查不标记）。
3. 发送成功后**再次** `check_and_mark_notify(mark=True)` 落标记——即"发送成功才记幂等"，失败不占用窗口。
4. 去重默认 TTL（`base.py` 硬编码，可被 policy 覆盖）：daily_report/premarket_outlook/tradingagents 12h、chart_analyst 6h、news_digest/兜底 60min、intraday_monitor 30min（intraday 另有按股票节流）。
5. `NotifierManager.notify_with_result()`：静默判断 → sanitize（Telegram/Bark/Pushover 走纯文本，其余走 Markdown 原文）→ 钉钉关键字自动追加 → Apprise 渠道与自定义渠道各自独立重试（指数退避 `backoff * 2^(i-1)`）→ 汇总错误返回 `{"success": bool, "error"/"skipped": ...}`。

### Telegram 的两个特殊处理

- 配置代理（渠道级 `proxy` 或全局 `http_proxy` AppSettings）时 `build_apprise_url` 返回 `None`，降级到自定义 `_send_telegram`（httpx + AsyncHTTPTransport(proxy=...)）。
- 自定义发送走老 Markdown parse_mode：全文先剥成纯文本，仅标题用单星号加粗；超 3900 字符截断，并优先把末尾裸 URL 保留为"完整报告"链接。

### 调度器全景（均在 `server.py` lifespan 启动）

| 调度器 | 注册名 | 触发方式 |
|---|---|---|
| `AgentScheduler`（`scheduler.py`，范围外） | — | 每个 enabled Agent 按 `cfg.schedule` 注册 cron/interval 任务；`reload_scheduler()` 全量重建。 |
| `PriceAlertScheduler` | `price_alert` | interval 60s（jitter 20s） |
| `PaperTradingScheduler` | `paper_trading` | interval 60s（jitter 20s）+ cron 09:00/15:30 |
| `ContextMaintenanceScheduler` | `context` | interval 6h + cron 04:15 清理 + 三个 cron 刷新机会池 + 启动补跑 |

所有 interval 任务都带 `jitter`、`coalesce=True`、`max_instances=1`，且各 job 内部还有 `_running` 标志防重入——这是**多调度器共写一个 SQLite** 的错峰设计。`ContextMaintenanceScheduler` 的机会刷新 cron 时间以 UTC 写死（1:15/5:30/14:00），对应北京时间，依赖构造时传入的 timezone 为 UTC。

### 状态存储位置

- **DB 表**（`src/web/models.py`，SQLite at `DATA_DIR`）：`notify_channels`（渠道配置+is_default）、`app_settings`（notify_* / pt_notify_* / http_proxy）、`notify_throttle`（节流+通知幂等共用）、`agent_runs`、`analysis_history`（含 `agent_kind_snapshot`）、`paper_trading_*`、`strategy_signal_runs`。
- **内存**：`scheduler_registry._REGISTRY`、`NotifierManager` 的渠道列表与钉钉关键字集合（每次发送前重建，见下）。
- **JSON 文件**：本范围内无。

## 对外接口

- **API 路由**：`src/web/api/channels.py`（渠道 CRUD/测试）、`settings.py`（`notify_*` 设置项，有描述文案）、`agents.py`（`preview_schedule`/`count_runs_within` 暴露为调度预览接口）、`paper_trading.py`（`pt_notify_*` 设置 + 测试通知，调用 `send_test_notification()`）。
- **注册点**：`server.py` lifespan 创建并 start 四个调度器；`scheduler_registry.register()` 供自检；Agent 通知路径经 `server.py::resolve_notify_channels()`（stock_agent 级 > agent 级 > 默认渠道，按优先级覆盖 channel_ids）。
- **配置项**：env（`src/config.py`）：`notify_quiet_hours`、`notify_retry_attempts=2`、`notify_retry_backoff_seconds=2.0`、`notify_dedupe_ttl_overrides`、`app_timezone`；AppSettings 同名词覆盖 env。模拟盘通知独立一套 `pt_notify_enabled / channel_ids / realtime / premarket / summary`（默认关）。

## 二次开发扩展点与注意事项

1. **新增通知渠道**：改 `CHANNEL_TYPES`（表单字段）+ 决定走 Apprise 还是 `_send_custom`；若是 Markdown 渠道加进 `_MARKDOWN_CHANNELS`，否则加 `_PLAIN_TEXT_CHANNELS`。钉钉加签/@手机走 URL 约定（`dingtalk://secret@token/?to=`）。前端表单字段需同步。
2. **NotifierManager 是短生命周期对象**：`paper_trading_notifier._build_notifier()` 每次发送都新建 manager、每次 `_load_config()` 都开新 DB session（一条消息内会重复读配置 3 次）。改配置即时生效，但不要往 manager 上挂需要跨调用保留的状态。
3. **cron day_of_week 语义陷阱**：UI/DB 里是 POSIX（0/7=周日），APScheduler 是 0=周一。新增调度入口必须走 `parse_schedule()`，不要直接 `CronTrigger.from_crontab`。
4. **`notify_throttle` 表被两处复用**：intraday 按股票节流（scope=股票代码）和全局通知幂等（scope=`__notify__:<sha1>`）写同一张表，改表结构或清理逻辑要同时考虑两者。
5. **幂等去重的 fail-open 语义**：`check_and_mark_notify` 异常时返回 True（宁可重发不可漏发）；只有发送成功才 mark。若改成"发送前标记"，重启/并发下会漏通知。
6. **调度器时区**：`AgentScheduler` 用 `settings.app_timezone`；三个内置调度器由 server.py 传 timezone（需确认传值，机会刷新的 UTC 小时注释依赖该值为 UTC）。`PaperTradingScheduler` 的盘前 09:00/日终 15:30 是按调度器时区解释的——改 timezone 会平移这两个 cron。
7. **新增内置调度器**：记得 `start()` 里 `scheduler_registry.register()`，否则自检看不到；shutdown 用 `wait=False`。
8. **`analysis_history` 的覆盖语义**：同日同 agent+symbol 会覆盖写；`get_latest_analysis` 是严格 `analysis_date < before_date`（不含当天），要含当天用 `get_latest_ta_verdict_row` 的 +1 天技巧。`agent_kind_snapshot` 在保存时从 `agent_catalog.infer_agent_kind` 快照，改分类逻辑不影响历史行。
9. **`agent_runs` 字段截断**：result/error 只存前 2000 字符、trace_id 64、trigger_source 32——排查问题要看日志而不是只看表。
10. **SQLite 并发写**：60s 级扫描任务的 jitter/防重入标志是有意为之，新增高频任务沿用同样模式，不要去掉 jitter。
