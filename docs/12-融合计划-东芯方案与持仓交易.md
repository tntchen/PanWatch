## 0. 协作与审计约定（用户明确要求，全程有效）

1. 本项目为**中等复杂度二次开发**，采用**阶段门禁制**：每个 Phase 开工前必须经用户明确确认；每个 Phase 完成后提交变更清单供用户审计，确认后才进入下一阶段。
2. 计划文档（本文件）是唯一事实来源，任何方案调整先改本文件、经用户确认后再动代码。
3. 决策点（第 5 节）逐条确认，确认结果回填到状态表；未确认的项按"待定"处理，不得擅自按推荐值开工。
4. 开发过程中发现计划外的新问题/新选项时，暂停并回报，不自行扩大改动范围。
5. **fork 冲突面原则**：新文件优先于改旧文件；必须改旧文件时把增量压到最小并记录在本文件。

## 1. 两个目标

1. **持仓交易化**：持仓页支持加仓 / 减仓 / 清仓，有交易流水，成本自动重算（现在只能手动改总数、自己算加权成本）。
2. **个股方案跟踪**：把 Kimi Chat 里"深度研究 → v3.1 方案 → 每日定时跟踪 → 备忘录归档"的闭环搬进 PanWatch。**做成通用能力（任何股票都能建方案档案），东芯作为第一个实例录入**。

**边界声明（v1.6）**：本期融合的是「方案档案 + 跟踪闭环」。Kimi 的**研究流程**（8 维调研 → 交叉验证 → 47 页报告写作）**不在本期范围**（P5 远期项）——迁移后的分工是：研究仍在 Kimi 完成 → 方案档案录入 PanWatch → PanWatch 接管每日跟踪。若要将研究流程也迁入，需另行立项（且需先设计与现有 TradingAgents 的关系）。

## 3. 分期计划

### Phase 1 — 持仓交易化（加仓/减仓/流水）

**语义约定（评审 C3，开工前已定死）**：
1. 成本口径：移动加权平均；**计算用 Decimal，落库 float**，测试断言写容差。
2. 加仓：重算加权成本；减仓：成本不变、按卖出比例结转已实现盈亏、`invested_amount` 同比例结转。
3. 关仓 = quantity=0 **保留行**（不删行，流水不被级联清空）；有流水的持仓**禁止删除**。
4. 手动编辑成本/数量降级为自动生成 **adjustment 流水**（杜绝双写）；重新买入已关仓股票走 trades 端点。
5. 「流水插入 + 成本重算 + Account 资金联动」单事务提交。

**数据模型**：新表 `position_trades`（position_id、direction、price、quantity、fee、traded_at、realized_pnl、备注），参照 `PaperTradingTrade`。**新表走 create_all，不用版本化迁移**；历史流水（1600→3150 三笔）用幂等导入脚本或手工走 API 录入，绝不写进迁移。

**后端**（`src/web/api/accounts.py`）：`POST /positions/{id}/trades`、`GET /positions/{id}/trades`；`portfolio/summary` 增加已实现盈亏。

**前端**：持仓行加「加仓/减仓/流水」；**流水弹窗抽独立组件**（Stocks.tsx 不再增长）；注意前端在 `Stocks.tsx:306/324` 本地重算盈亏，需同步改类型定义 + 重算逻辑 + 汇总卡。

**测试**：成本重算纯函数单测（加仓/减仓/归零/手续费/adjustment）+ API sqlite 测试；`test_portfolio_*` 回归。

**验收**：录入三笔历史流水后成本 = 112.572（容差内）；减 500 股成本不变、已实现盈亏正确、资金联动；关仓保留行且禁删；再建仓走 trades 端点。

### Phase 2 — 方案档案 + 触发器扩展（拆 P2a / P2b，可并行）

**P2a. 个股方案档案（stock_playbooks）**
- 新表 `(stock_id, version, is_active, payload JSON)`，payload 内含 `schema_version` 信封，演进只增不改、loader 容错读取（create_all 建表）。
- 新文件 `src/core/playbook.py`：loader（读 active 版本，无档案返回 None）+ **摘要生成器**（≤500 token/股：价位表/批次状态/临近日历项/策略模式）。
- UI：股票详情页「方案」Tab（结构化摘要 + 原始 markdown）。
- 东芯 v3.1 档案作为第一条数据录入。

**P2b. 触发器扩展（混合方案，评审 C2 定案）**
- `turnover_rate`：即时值，进现有引擎（白名单 + 取值约 3 行）——可作为**独立快赢最先落地**。
- `capital_flow`：走 CapitalFlowCollector 600s TTL 缓存，CN-only。
- `consecutive_close`：取最近 N 根日 K 收盘价**无状态评估**（复用 KlineCollector 缓存与 volume_ratio 回退先例），不存跨日状态、不新建调度器。
- 全部新条件 **CN-only 双层门控**（引擎 + 表单）；前端条件字典共三处：`PriceAlerts.tsx:74`、`biz-ui/price-alert-form-dialog.tsx`、`biz-ui/stock-price-alert-panel.tsx:53`。
- `price_alert_rules` 加可空列 `playbook_id`（**此项走版本化迁移 v119**，不修改已发布迁移函数），命中通知带方案提示文案；引擎 `_send_notify` 注入统一 NotifyPolicy（静默时段生效）。
- 测试参照 `test_price_alert_volume_ratio.py` 模式。

**验收**：东芯 7 条核心触发器全部可配置、可 dry-run；港美股规则配新条件被明确拒绝/提示。

### Phase 3 — 定时跟踪（P3a→P3b→P3c→P3d；P3a 只依赖 P2a，可与 P2b 并行）

**P3a. 档案注入公共层（评审 M1 定死；v1.5 扩充流水注入）**
- 在 `ContextBuilder.build_symbol_contexts()` 的 payload 加 `"playbook"` 键（值= P2a 摘要，无档案为 None）——对上游 1-2 行增量，premarket/intraday/daily_report 三个消费方一次接入。
- **流水注入（v1.5 新增，用户要求）**：各 Agent 的持仓上下文从"仅数量/成本"升级为"数量/成本 + 近期交易流水"。实现：扩展 `load_portfolio_for_agent`（server.py:774）的 PositionInfo 附带近期流水（默认当日全部 + 近 10 笔，字段：日期/方向/价/量/费/已实现盈亏/备注），prompt 中以紧凑文本呈现（≤200 token/股）。
- 消费规则：
  - **daily_report（收盘复盘）**：自动对照当日流水与方案纪律，点评执行（替代 Kimi 时代"用户告知操作→先更新档案再复盘"的人工环节）；持仓盈亏用流水精确计算。
  - **tail_brief（尾盘决策）**：给出执行细节时考虑当日已有操作（如当日已做 T 减出则提示接回区）。
  - **morning_brief / 盘中监测**：感知近期补仓/减出节奏，避免与近期操作矛盾的建议。
  - 无流水的持仓行为不变（向后兼容）。
- 原则：**注入摘要非全文**；有档案走方案逻辑、无档案行为完全不变。
- 测试：prompt 组装断言（有档案出现摘要、无档案 prompt 不变；有流水出现交易记录、无流水 prompt 不变）。

**P3b. 收盘复盘（daily_report）改版（评审 C1 降级后）**
- 时间 seed 默认 `30 15` → `0 18`；**前置验证：实测东财龙虎榜 18:00 数据是否出齐**，不出齐则保留 15:30 + 夜间补一轮。
- 复用现有 `md.dragon_tiger()` 通道（必要时加 host 侧 accessor `md_dragon_tiger()`），按个股过滤；无备源，**失败容错跳过该章节**。
- 有档案股票增加：✅ 触发器逐项状态 → 💰 持仓盈亏（用 P1 流水）→ 📅 次日预案 → 🗓 日历提醒。
- 注意：seed 不覆盖现有部署 schedule，上线 checklist 含「老库手动 UI 改时间」。

**P3c. 盘中监测（intraday_monitor）套用方案逻辑**
- 事件门接入方案价位（做 T 区/防线/右侧批触发区穿越即事件）；有档案股票可用方案参数覆盖事件门阈值；prompt 带档案摘要。

**P3d. 新增 morning_brief / tail_brief 两个 workflow Agent（评审 M6 规则）**
- `morning_brief`：`0 10 * * 1-5`；`tail_brief`：`45 14 * * 1-5`；均 single 逐股。
- **默认仅对有档案股票启用**（避免与盘中监测同源重复）。
- `run_single` 用**传参式**实现（不复用"临时改共享 watchlist"的不安全模式）。
- 注册：`AGENT_REGISTRY` + `AGENT_SEED_SPECS` + `WORKFLOW_AGENT_NAMES` 元组 + `prompts/` 两个模板 + base.py 去重 TTL 表加两行；建议词表与 `*_ACTION_MAP` 同步设计。
- 调度走 AgentScheduler，自动被 selfcheck 覆盖；确认 `max_instances=1` + 错峰分钟。

**P3e. 数据补齐（v1.6 新增，依据 docs/15 核查；建议在 P3b/P3d 消费前完成）**
- 科创50 日 K：`INDEX_SECID` 加 `"000688": "1.000688"`（client.py:36，一行）。
- 融资余额周度序列：margin vendor 返回全部 30 行而非 rows[0]（market_flow.py:146-149）。
- 存储板块定向行情：补东财板块 quote/K线端点（secid=90.BKxxxx）或调用侧先 hot_boards 定位板块代码。
- 日 K 成交额：kline vendor fields2 加成交额字段映射进 Bar（"缩量企稳"可按成交额判断）。
- 均为 packages/marketdata 包内小改，按规范补包内测试。

**验收**：早盘/尾盘对有档案股票自动生成模板简报并推送；18:00 复盘含龙虎榜解读（或容错跳过说明）、触发器状态与持仓盈亏；盘中穿越方案关键位必触发；**当日有流水时收盘复盘含"当日操作 vs 方案纪律"点评（v1.5）**；P3e 四项数据在简报组装链路可实际取到（v1.6）。
