## 1. JWT 设计（T16 / T20）

### 1.1 claims 结构

| claim | 值 | 说明 |
|---|---|---|
| `sub` | `str(user.id)` | 用户主键。废除 F2 的字面量 `"user"`。JWT 规范 sub 为字符串，统一 `str()` 编码/解码后 `int()` |
| `tenant_id` | `int` | **身份穿透的核心 claim**。与 `users.tenant_id` 一致；签发时从库取，不在客户端可改 |
| `role` | `"admin" \| "user"` | T5 两级。注意：T5 明确 **role 实时查库**，token 内 role 仅作展示/快速判定，权限裁决以 `get_current_user` 查库为准（见 §2.4） |
| `iat` / `exp` | UTC 时间戳 | 保留现有语义（`auth.py:87-89`） |
| `pwd_at` | `int`（密码最后修改时间戳） | **吊销能力**：校验时若 `user.password_changed_at > pwd_at` → 401。解决 F6「改密后旧 token 仍有效」，无需 refresh token 体系 |

`sub`/`tenant_id`/`role` 三个 claim 均为**必填**，缺失即拒绝（见 §1.3，这同时实现旧 token 一次性踢出）。

### 1.2 签发/校验改造点

| 函数 | 现状 | 改造设计 |
|---|---|---|
| `create_token`（`auth.py:83-93`） | 无参、sub 硬编码 | 签名改为 `create_token(user: User) -> tuple[str, datetime]`，payload 按 §1.1 填充 |
| `verify_token`（`auth.py:96-104`） | 返 bool | 改为 `decode_token(token) -> TokenClaims \| None`，解出并校验全部必填 claim 与类型 |
| `get_current_user`（`auth.py:160-186`） | 返字面量 | 返回 `CurrentUser(user_id, tenant_id, username, role)` dataclass；**实时查库**（T5）：每请求按 `sub` 查 users，用户被删/禁用/改密（比对 `pwd_at`）即 401。性能：单行主键查询，N≤5（T1）无压力 |
| 26 个 protected router（`app.py:64-185`） | 路由级 Depends | **保持路由级 Depends 不变**——`get_current_user` 在依赖内设置 TenantContext（§3.2），handler 仍零改动。这是 docs/17 §5「前端改动小」结论成立的前提 |

### 1.3 旧 token 一次性踢出（T20 / docs/20 M10）

**结论：用 claim schema 严格化实现，零额外状态，优先于密钥轮换。**

1. 上线后所有旧 token 的 payload = `{exp, iat, sub:"user"}`（F2），缺 `tenant_id`/`pwd_at` → 必填 claim 校验直接 401。天然踢出，无兼容期（M10「不设兼容期」）。
2. 密钥轮换作为**冗余手段**而非主手段：`jwt_secret` 维持存于 `app_settings`（T20 判定其实例级；现值由 `get_jwt_secret` 生成，`auth.py:37-60`）。不建议上线时强制轮换——claim 校验已足够，轮换会让「踢出原因」不可观测。
3. 日常吊销能力由 `pwd_at` claim 提供（§1.1），改密/管理员重置密码 → 该用户全部旧 token 失效。

### 1.4 有效期建议

**建议：维持 30 天不变（`JWT_EXPIRE_DAYS = 30`，`auth.py:22`）。** 理由：

- T1 确认 ≤5 人私有部署，无 refresh token 基础设施，缩短有效期=纯体验损失；
- 真正的吊销缺口（改密后旧 token 有效）已由 `pwd_at` 补齐；
- 安全收益/成本比最高的动作是踢出机制（§1.3）而非缩短过期。

如未来开放外网访问再引入 refresh token 双 token 体系，MT-P0 不设计。

## 2. users / tenants 模型与认证面（T5 / T12 / T20）

### 2.1 表结构（v120 自包含建表，R4）

```text
tenants:  id INTEGER PK | name TEXT NOT NULL | created_at DATETIME
users:    id INTEGER PK | username TEXT NOT NULL UNIQUE | password_hash TEXT NOT NULL
          role TEXT NOT NULL DEFAULT 'user'        -- 'admin' | 'user'（T5 两级）
          tenant_id INTEGER NOT NULL DEFAULT 1 REFERENCES tenants(id)  -- T18
          is_active INTEGER NOT NULL DEFAULT 1
          password_changed_at DATETIME             -- 对应 pwd_at claim
          last_login_at DATETIME | created_at DATETIME
```

- v120 内 `INSERT INTO tenants(id, name) VALUES (1, '默认租户')` + 初始管理员（§2.3），自包含不依赖 create_all（R4/M3）。
- `tenant_id NOT NULL DEFAULT 1` 与全库 T18 策略一致，611 基线透明（docs/20 C5）。
- SQLite 限制遵守：users 是新表不受「ADD COLUMN 不能 NOT NULL DEFAULT+FK」限制；旧表加列在 v120 由 schema 文档负责，本文不展开。

### 2.2 bcrypt 透明重哈希（T20 / docs/20 M10）

- 依赖：`requirements.txt` 新增 `bcrypt`（F15，当前无哈希库）。不用 passlib（维护停滞）。
- 哈希格式判别：bcrypt 串以 `$2b$` 开头；旧格式 = 64 位十六进制（`hashlib.sha256(...).hexdigest()`，`auth.py:78-80`）。
- **登录流程（替换 `auth.py:218-233`）**：

```text
1. 按 username 查 users；不存在 → 401「用户名或密码错误」（不区分用户不存在/密码错）
2. if password_hash 以 "$2b$" 开头 → bcrypt.checkpw
   elif 全长 64 hex → sha256(输入) == 存储值
        → 校验通过即：password_hash = bcrypt.hashpw(输入)，password_changed_at = now，写回（透明重哈希）
   else → 401（未知格式按失败处理）
3. 校验通过 → 更新 last_login_at → create_token(user)
```

- **不可逆性说明**：旧 SHA-256 无法离线转 bcrypt（M10 原文确认），只能首次登录时重哈希；未登录过的账号保持旧格式直到首次登录。管理员重置密码直接写 bcrypt。
- `/setup`（`auth.py:198-215`）与 `/change-password`（`auth.py:236-249`）新写的哈希一律 bcrypt。

### 2.3 废除「未设密码全放行」+ env 凭证迁移 + 邀请制

| 项 | 现状 | 设计 |
|---|---|---|
| 全放行 | `auth.py:166-169` 未设密码 return None | **删除该分支**。`get_current_user` 无条件要求合法 token。这是 docs/17 §1.1-3 的硬性前置 |
| bootstrap | `/setup` 任何人可首设 | 保留语义但改为「**仅当 users 表 0 行时可用**」，创建的是 tenant=1 的 **admin**；`app_settings` 的 `auth_username`/`auth_password_hash` 两行废弃（T20：auth_* → users 表），v120 迁移把旧 KV 转为 users 行（username 原值、sha256 哈希原样搬入待透明重哈希） |
| env 凭证 | `init_auth_from_env`（`auth.py:141-157`，`server.py:1546-1553` 调用）写 app_settings | 改为写 users 表：`AUTH_USERNAME` → 初始 admin（tenant=1），`AUTH_PASSWORD` 直接 **bcrypt 哈希后入库**（env 是明文，无需走 sha256 兼容路径）；"已有账号不覆盖"守卫（`auth.py:151-152`）改为「users 表非空即跳过」 |
| 注册 | 不存在 | T12 邀请制：**不开公开注册端点**。管理员端点 `POST /api/auth/users`（仅 admin）：入参 username+初始密码+tenant 选项（T13：「与管理员共享配额」标志写 users 扩展列，属 T13 设计文档范围，本文仅预留）。MT-P1 上线即只有默认 admin（docs/20 M1：P1→P2 禁止多用户） |
| 禁用/删除 | 不存在 | `is_active=0` 即封禁；`get_current_user` 实时查库（§1.2）使封禁即时生效，无需等 token 过期 |

## 3. contextvar TenantContext + `do_orm_execute` 自动过滤（T16 / T18 / docs/20 C7）

### 3.1 模块与载体

新模块 `src/web/tenant_context.py`：

```python
_tenant_ctx: ContextVar[TenantContext | None]  # TenantContext(user_id, tenant_id, role, is_background)

def set_tenant_context(ctx) -> Token      # 返回 token 供 reset
def get_tenant_context() -> TenantContext | None
def require_tenant_id() -> int            # 无上下文 → raise TenantContextMissingError
@contextmanager
def tenant_scope(tenant_id: int, *, background: bool = True)  # 后台任务显式传参入口
```

### 3.2 request 生命周期（设置/清理）

| 时点 | 动作 | 依据 |
|---|---|---|
| 请求进入 | 新增 Starlette 中间件（加在 `ResponseWrapperMiddleware` 之前，`app.py:49`）：`_tenant_ctx.set(None)`——**每请求强制清空**，防 threadpool 线程复用导致的上请求残留（FastAPI sync handler 跑 anyio threadpool，contextvar 随线程不随请求） | F17 引擎全局单例；线程复用风险 |
| 鉴权通过 | `get_current_user`（依赖，`app.py:64` 全 protected router 收口）内 `set_tenant_context(...)` | 26 个 router 零改动（§1.2） |
| 公开路由（/api/auth、/api/market、/reports、`app.py:59-61, 206`） | 不设置；这些路由不得触达私有表，由 default-deny（§3.4）兜底 | F9 |
| 请求结束 | 中间件 `finally` 无需显式 reset——下一请求开头已强制清空；但后台线程（§3.5）必须显式 reset | — |

### 3.3 `do_orm_execute` 事件与表的三分判定

在 `src/web/database.py`（`SessionLocal` 定义处，`:38`）挂 SQLAlchemy 2.0 事件：

```python
@event.listens_for(Session, "do_orm_execute")
def _apply_tenant_filter(execute_state):
    # 仅处理 SELECT；对 REGISTERED_PRIVATE_MODELS 逐个叠加
    # with_loader_criteria(Model, lambda cls: cls.tenant_id == ctx.tenant_id, include_aliases=True)
```

**判定规则（default-deny，按序匹配）**：

| 优先级 | 规则 | 动作 |
|---|---|---|
| R1 | 语句涉表 ∈ **共享表白名单**（无 tenant_id 列：`data_sources`、`news_cache`、`market_*_snapshots`、`strategy_*`、`factor_*`、`schema_migrations` 等，清单归 schema 文档 T19） | 跳过过滤 |
| R2 | 语句涉表 ∈ **私有表注册表**（有 tenant_id 列且已登记） | 有 ctx → 自动加 `tenant_id == ctx.tenant_id`；**无 ctx → raise TenantScopeError**（拒绝，fail-closed） |
| R3 | 语句涉表**有 tenant_id 列但未登记** | 一律 raise TenantScopeError（default-deny：新表/漏登记第一时间爆炸而非静默泄漏，docs/20 C7） |
| R4 | INSERT/UPDATE/DELETE | ORM 事件同样拦截：UPDATE/DELETE 的 WHERE 叠加 tenant 条件；INSERT 由模型列默认值 + 写入侧 helper（§4）保证。`bulk_insert_mappings`（如 `log_handler.py:120`）**不触发 ORM 事件**，属 §4.2 显式处理范围 |

注册表 = 单一事实源：`TENANT_SCOPED_MODELS: list[type[Base]]`，schema 文档每确认一张私有表即登记一张；未登记即 R3 拒绝，形成「登记 = 审计完成」的机械对账。

### 3.4 单租户回退 flag（T20 / docs/20 M14）

环境变量 `PANWATCH_SINGLE_TENANT=1`：

- `get_current_user` 正常鉴权（认证不关闭，关闭的是"多"），TenantContext 恒为 tenant=1 默认管理员；
- `do_orm_execute` 过滤条件退化为常量 `tenant_id == 1`（与 T18 默认值一致）；
- 管理员端点（邀请/建租户）隐藏；
- 验收 = 611 基线全绿（M14），见 §7.3。

### 3.5 后台任务（无 request 上下文）显式传参约定

- 调度器/引擎/agent 链：**禁读 contextvar 当作身份来源**；统一入口 `with tenant_scope(tenant_id):` 包裹每个租户的遍历体（配合 T17 单 job 遍历租户硬约束，docs/20 C8）。`tenant_scope` 内 `do_orm_execute` 照常自动过滤——后台与 web 复用同一机制，但 ctx 来源是显式参数而非 token。
- `tenant_scope` 必须 `try/finally reset(token)`，防调度线程串租户。
- `context_builder(tenant_id, agent_name)` 签名重构属另一设计文档，本文只约定：**任何自建 `SessionLocal()` 的代码路径要么在 `tenant_scope` 内，要么触发 R2/R3 拒绝**——机制保证漏传即爆炸。

## 6. `/reports` 签名 URL（F9 / T20）

### 6.1 设计

- 现状：`GET /reports/{filename}` 无鉴权（`app.py:206-227`），防穿越已完备（basename + realpath 双保险，`:212-224`）；报告下载链接随通知外发（`panwatch_base_url` 拼接，`analysis_link.py:38-47`），要求免登录可达。
- 方案：**HMAC 签名查询参数**，不加登录态：

```text
/reports/{tenant_id}/{filename}?exp=<unix_ts>&sig=<hex>
sig = HMAC_SHA256(key=jwt_secret, msg=f"{tenant_id}|{filename}|{exp}")
```

- 校验：`download_report` 增加——exp 过期 → 410；sig 不匹配 → 403；通过后按 `{REPORTS_DIR}/{tenant_id}/{filename}` 取文件（目录按租户隔离，属 MT-P2 迁移范围，本文只定路由形态）。
- 签名 key 复用实例级 `jwt_secret`（T20 实例级），不引入新密钥管理面。
- **有效期：7 天**。报告是时点产物，7 天覆盖通知点击窗口；过期后用户登录系统从报告页重取（届时生成新签名链接）。
- 生成侧：通知链拼报告链接处（T21 按 tenant 路由后）调用统一 `make_signed_report_url(tenant_id, filename)`。

### 6.2 分期兼容

| 阶段 | 行为 |
|---|---|
| MT-P1（认证先行） | **保持现状无鉴权**（M1 窗口期本就"有认证无隔离"，此时收紧 /reports 只打断通知无安全收益） |
| MT-P2（数据隔离 + reports 目录按租户拆分） | 启用签名校验。存量无签名旧链接一次性失效（与旧 token 踢出同风格，docs/20 M10「不设兼容期」）；`PANWATCH_SINGLE_TENANT=1` 回退 flag 下 sig 缺失时按 tenant=1 放行 |
| MT-P5 | 验收：伪造 sig、过期 exp、跨租户 filename（A 的 sig 取 B 的文件）三用例必过 |
