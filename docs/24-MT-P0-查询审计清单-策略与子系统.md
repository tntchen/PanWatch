### 2.3 entry_candidates 按 candidate_source 拆语义方案（T19 核心交付）

**现状语义**（`entry_candidates.py:1276,1293,1394`）：`candidate_source ∈ {market_scan, watchlist, mixed}`；UQ = (stock_symbol, stock_market, snapshot_date)（`models.py:501-506`），单用户下天然一行。

**拆分判定**：

| source | 输入来源 | 行内容是否含租户文本 | 判定 |
|---|---|---|---|
| `market_scan` | 市场池扫描（`_load_market_scan_inputs` :977，市场数据驱动） | 否（`_derive_market_scan_decision` :627 模板化文本） | **市场级行**，tenant_id=0 |
| `watchlist` | 本租户 StockSuggestion（:1259 `:1286`） | **是**（reason/signal 源自租户 AI 输出） | **租户行**，tenant_id=租户 |
| `mixed` | 市场池 ∩ 租户 watchlist（:1276,1293） | **是**（合并了 suggestion 文本） | **租户行**，tenant_id=租户；市场级视图由同日 market_scan 行独立提供 |

**表结构方案（推荐方案 A，哨兵行）**：
1. `entry_candidates` 加 `tenant_id INTEGER NOT NULL DEFAULT 1`（T18 回填语义：存量全=1）；**市场级行写 tenant_id=0**（文档化哨兵，不进 tenants 表，无 FK——规避 R4 的 SQLite ADD COLUMN 限制）。
2. v121 重建 UQ → `(tenant_id, stock_symbol, stock_market, snapshot_date)`。
3. 写入侧拆分（T17 单 job 内顺序执行）：全局 pass 只产 market_scan 行；随后遍历租户，每租户用本租户 suggestions+holding_keys 产 watchlist/mixed 行。`refresh_entry_candidates` 需加 `tenant_id: int | None` 参数（None=全局 pass）。
4. 读取侧：T16 混合表过滤 `tenant_id IN (0, current)`；`list_entry_candidates` 的 `source=market_scan` 分支天然只命中 tenant_id=0 行（:1587-1590 语义保留）。
5. 级联：entry_candidate_outcomes 的 tenant 从父 candidate 派生（T18 不变量，对账 SQL 进 v122）；entry_candidate_feedback 无父 FK，**tenant_id 直接落列**（写入点 `entry_candidates.py:1645-1653` 需加参数；API 写入点 `feedback.py:23` 从 TenantContext 取）。

**方案 B（备查，不推荐）**：market_scan 行按租户复制——存储 ×N、刷新 ×N，违背 T19 全局本意与 N≤5 虽可行但无谓浪费。
