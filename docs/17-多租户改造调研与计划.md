## 1. 总体结论（一句话版）

当前系统是**彻底的单用户设计**：没有 users 表，JWT 里 `sub` 是硬编码字面量 `"user"`（`src/web/api/auth.py:90`），鉴权只在路由边界当门禁、**身份随即被丢弃**（25 个 router 用路由级 Depends，全代码库零 handler 接收用户对象）。多租户改造的本质不是"加 tenant_id 列"，而是**建立身份传递通道 + 全量查询审计 + 调度链穿透**三件事。行情数据层是天然共享红利（几乎零改造），真正的硬骨头是三处"静默串扰"。

### 1.1 单用户假设最深的五个根

| # | 根 | 证据 |
|---|---|---|
| 1 | 无 users 表，凭证是 app_settings 三行 KV（`auth_username`/`auth_password_hash`/`jwt_secret`） | `src/web/api/auth.py:29-31` |
| 2 | token 无身份：`create_token` 硬编码 `sub="user"`，`verify_token` 只返 bool，`get_current_user` 返回字面量 `"user"` | `auth.py:83-104, 186` |
| 3 | **未设密码全站放行**（多租户上线前必须废除，否则全租户裸奔） | `auth.py:166-169` |
| 4 | 调度器/告警引擎/模拟盘引擎全是进程级单例，遍历全表无用户维度 | `server.py:50-54`、`src/core/price_alert_engine.py:617`、`src/core/paper_trading_engine.py:240-241`（模拟盘 `.first()` 全库单账户） |
| 5 | 44 张业务表中约 30 张无任何属主维度；21 处唯一约束编码了"全局唯一" | `src/web/models.py` 逐表盘点见 §2 |

### 1.2 三大"静默串扰"硬骨头（不报错、测试最难发现，优先处置）

1. **通知去重/节流互吞**：`NotifyThrottle` 的去重键 = `agent_name + 内容hash`（`src/agents/base.py:269-328`、`src/core/notify_dedupe.py:17-25`）。两个租户自选股相同 → hash 相同 → **后发的租户通知被当重复吞掉**，无任何报错。intraday 个股节流（`intraday_monitor.py:943-1008`）同理。
2. **事件门文件级状态**：`data/state/intraday_monitor_state.json` 按 symbol 单文件存 `last_price` 基线与冷却（`src/core/intraday_event_gate.py:22-27, 251-342`）。多租户交替写 → 穿越检测错乱、冷却互相消耗。不在 DB 里，**表结构迁移必然漏它**。
3. **调度 job 命名 + 穿透链**：job `id=agent.name` + `replace_existing=True`（`src/core/scheduler.py:51-58`）——"每租户注册一遍 agent"会导致只有最后一个租户在跑；且租户参数要穿透 `scheduler → context_builder → build_context → 4 个 load/resolve → AgentContext` 的长链，漏一环就静默用错数据。

### 1.3 多租户红利点（保持共享，切勿按租户复制）

- **marketdata 包 + collectors 缓存**：零用户概念，TTL 缓存按 symbol key（`packages/marketdata/src/marketdata/cache.py`）——N 个租户盯同一只股只产生一份上游请求，租户越多缓存命中率越高。
- **市场级快照与策略计算**：`StockContextSnapshot`/`NewsTopicSnapshot`/`market_regime_snapshots`/`market_scan_snapshots`、上下文维护调度器（策略信号/因子校准，`src/core/context_scheduler.py`）——全实例一份即可。
- **新闻缓存/全市场股票清单**：`news_cache`、`data/stock_list_cache.json`。
- 部署面：单库方案下 Docker 镜像/CI/Watchtower/端口 8000 约定全部可原样保留。

## 2. 数据层盘点（范围 A 摘要）

### 2.1 表的所有权分类（44 张业务表）

- **强私有（约 30 张，必须加 tenant_id）**：accounts/positions/position_trades/stocks/stock_agents/stock_playbooks/price_alert_rules/price_alert_hits/ai_services（含 api_key 明文）/ai_models/notify_channels（含 bot_token 明文）/paper_trading_*（注意：模拟盘账户是注释明写的"单例"，持仓/流水连 account FK 都没有）/chat_*/analysis_history/stock_context_snapshots/agent_context_runs/agent_runs/stock_suggestions/entry_candidates 及反馈表/notify_throttle/log_entries/strategy_signal 链。
- **混合语义（最难归类，需拍板）**：`agent_configs`（name 全局 unique 是系统 seed，但 enabled/schedule/ai_model_id 是用户偏好）、`app_settings`（认证凭证/jwt_secret/http_proxy/ui_avatar/notify_* 混装一个 KV 表）、`strategy_catalog`/`strategy_weights`/`factor_weights`（全局自动标定 vs 每租户标定是产品决策）。
- **天然全局共享**：data_sources/news_cache/market_*_snapshots/schema_migrations/stock_list_cache。
- **语义本身就错（多租户下第一个爆）**：`portfolio_risk_snapshots` 内容是用户组合风险，唯一约束却是 `UQ(snapshot_date, market)`（`models.py:906-910`）——单用户下碰巧正确，两租户同天同市场直接撞约束。

### 2.2 迁移机制的三个陷阱

1. **checksum 机制禁改旧迁移**：迁移 checksum = sha256(版本+名称+源码)，改 v101–119 任何一行源码会触发存量库重跑全部历史迁移（含 INSERT 回填），只能新增 v120+（`migrations.py:20-33, 1638-1645`）。
2. **双轨迁移并存**：database.py 有 5 个 ad-hoc 迁移 + migrations.py 版本化迁移；且新表走 create_all（如 stock_playbooks）不走迁移——tenant 列必须在模型定义和迁移两处同步，漏一处新旧库 schema 漂移。
3. **SQLite 改唯一约束只能重建表**（项目已有 `stocks__new` 重建先例，`database.py:507-541`）；约 21 处唯一约束需要纳入 tenant_id 重写——这是一次性大迁移，备份机制（`database.py:114-131`）可复用。

### 2.3 DB 外落盘状态（迁移脚本管不到，最易漏）

`data/state/intraday_monitor_state.json`（事件门，见 §1.2）、`data/avatars/`（全局单头像）、`data/reports/`（全实例一目录，下载路由无鉴权）、`panwatch.db.bak.*`（备份含全量明文密钥，多租户后备份本身是泄露面）。

## 3. API 与鉴权盘点（范围 B 摘要）

- **身份传递通道完全不存在**：`app.py:64` 的 `protected = [Depends(get_current_user)]` 只是门禁，28 个路由文件的 handler 全部不接收 user。改造 = 建立身份通道 + **逐条查询审计**（漏一条就是跨租户泄漏）。
- **无用户维度过滤的代表性端点**：自选股 CRUD（`stocks.py:181+`）、账户/持仓/流水（`accounts.py:242+`）、告警（`price_alerts.py:149+`）、方案档案（`playbooks.py:80+`）、模拟盘（`paper_trading.py:354` 等 4 处 `.first()`）、AI 对话（`chat.py` 上下文拼装读全部自选股/持仓——**LLM prompt 带别的租户持仓是最易漏的泄漏口**）、`DELETE /api/logs` 无条件清全表（`logs.py:208-212`）。
- **"排他性 update"陷阱**：`is_default` 切换（AI 模型 `providers.py:138,231`、通知渠道 `channels.py:54,70`）是全表 `update({is_default: False})`——加 tenant 后必须同步加 tenant 条件，否则租户 A 设默认清掉租户 B 的。
- **进程级单例的 API 引用点**：`agents.py:369`（trigger_agent）、`price_alerts.py:328`（告警调度）、`templates.py:265`（reload_scheduler）、`settings.py:197`（写进程代理 env）——函数体内延迟 import，静态搜索容易漏。
- **缓存型泄漏**：`accounts.py:780` 组合分析 TTLCache（key 不含租户会直接向租户 B 返回租户 A 的持仓市值，风险最高）、`agents.py:34-36` 盘中扫描缓存 key 只含 symbols。
- **/reports 下载**：无鉴权 + 文件名可枚举（`app.py:203-227`）；加鉴权会打断通知里的免登录外链（`panwatch_base_url`），需要签名 URL 方案。
- **密码安全债**：裸 SHA-256 无盐（`auth.py:78-80`）、token 30 天无吊销无 refresh、改密后旧 token 仍有效。

## 4. 后端运行时盘点（范围 C 摘要）

- **调度链**：4 个进程级调度器（agent/价格提醒/模拟盘/上下文维护）。`build_scheduler` 注册 `AgentConfig.enabled` 全量（`server.py:1220-1262`）；job id=agent.name（§1.2 第 3 条）。
- **数据加载无过滤**：`load_watchlist_for_agent`（`server.py:746`）、`load_portfolio_for_agent` 的 `Account.enabled==True` 全量拉取（`server.py:881`）。
- **"系统默认"解析**：`resolve_ai_model`/`resolve_notify_channels` 的 `is_default` 回退是全局唯一（`server.py:1045, 1106`）——需改为租户作用域。
- **种子逻辑**：`seed_agents`/`seed_data_sources` 按 name 全库 upsert + 删孤儿——多租户下需区分"系统模板 vs 租户行"，否则对账误删租户自定义。
- **TradingAgents 子系统**：5 处自建 SessionLocal 直查库（`tradingagents/agent.py:514` 等），绕过 build_context，各自需加过滤。
- **进程级代理**：`apply_proxy_env` 写进程 env + 全链路 `trust_env=True` 渗透 marketdata/collectors/notifier 三处 HTTP 栈——代理天然无法按租户，需显式定调为实例级。
- **SQLite 并发余量小**：现有代码已靠 jitter 错峰规避同刻写库（三处调度器注释）——租户数 ×N 后 60s 扫描的工作集与写竞争需重新评估。

## 5. 前端与部署盘点（范围 D 摘要）

- **好消息**：前端无 WS/SSE，鉴权只有一个收口点（`frontend/packages/api/src/client.ts:37-43` Bearer 注入），token 内嵌 tenant_id 后前端改动小；页面 URL 均无用户参数，数据过滤后端收口时组件大多不用改。
- **前端无"我是谁"**：登录后不保存不展示用户名，无 `/auth/me` 调用；AccountMenu 无身份显示（`AccountMenu.tsx:153-164`）。
- **命名碰撞**：前端 "Account" 已指持仓账户（`/accounts` 路由）、`PaperTradingAccount` 指模拟盘——引入用户概念时必须命名 `User`/`Tenant`，避免路由歧义。
- **localStorage 串味**：token/偏好/trace_id 约 10 个键是浏览器级单用户假设，同浏览器切账号会串（无安全风险，体验问题）。
- **部署形态是最根本分叉**：当前单容器+单卷+SQLite（`Dockerfile:99-121`、`deploy/docker-compose.yml`），Watchtower 追 latest 自动升级——多租户破坏式迁移会随镜像自动滚入生产，需加迁移门禁。

## 6. 方案选项

### 方案 A：共享库 + tenant_id 列（推荐）

单实例单库，全部私有表加 tenant_id，身份经 JWT claim 穿透。**部署面几乎不动**（镜像/卷/Watchtower/端口全保留），行情缓存红利全享。代价：约 30 表加列 + 21 处唯一约束重建 + 全量查询审计 + 调度链穿透，一次性大迁移。**适合：个人/小团队自用、几人共享、NAS 部署**。

### 方案 B：每租户一实例（多容器）

每租户独立容器+独立卷，代码改动最小（只需反向代理按域名分发 + 集中管理面）。代价：推翻现有 compose/Watchtower/NAS 自动化全套；每租户一套调度器重复请求上游行情（红利变负债）；N 个实例的升级/备份运维成本。**适合：强隔离合规要求、租户数少且付费**。

### 方案 C：混合（A 为主，敏感凭证实例级托管）

在 A 基础上，AI key/数据源凭证/通知 bot token 不做租户私有，改为**实例管理员统一托管**（租户只选不看见密钥）。大幅缩小权限模型与泄露面，最贴合"朋友共用我的部署"场景。

**Orchestrator 倾向：A + C 组合**——共享库 tenant_id 隔离业务数据，凭证类由管理员托管，权限只分"管理员/普通用户"两级。

# v1.1 修订（2026-07-22，按 docs/20 四角评审回填）

> 本节优先级高于 v1.0 正文。v1.0 原文保留作历史记录，冲突处以本节为准。

## R1. 事实勘误（评审核对代码后的更正）

1. protected router 是 **26 个**（非 25）。
2. tradingagents 直查库是 **6 个文件**（非 5）。
3. checksum 失配只**重跑失配那一条**迁移（非全部历史），且**无 INSERT 回填**——v120 发布即被 checksum 钉死不可热修，发布前必须演练。
4. 唯一约束 22 处定义，但仅约 **5–7 处需重建**：notify_throttle、analysis_history、stock_context_snapshots、portfolio_risk_snapshots、entry_candidates、新增 stocks 复合 UQ；FK 链传导的不必重建。
5. 查询审计面：全仓 `db.query(` 共 **455 处/59 文件**；路由层 201 处之外，运行时还有约 254 处（策略层约 100 处为最大审计簇）。
6. 备份机制**不可复用**：WAL 模式下 `copy2` 只拷主文件不拷 -wal/-shm，非一致性快照（修订见 R5）。
7. stocks 表无 symbol 唯一约束，全仓 **12 处** `Stock.symbol==X .first()` 解析点（清单见 docs/20 §2-M1/C6），多租户下随机命中别租户。
