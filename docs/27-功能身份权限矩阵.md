# 功能 × 身份权限矩阵（MT-P1 施工依据）

> 日期：2026-07-22 ｜ 阶段：多租户 MT-P1 前置核对
> 用户表示身份设置需求不明确、委托经验判断。本文把 T3/T5/T12/T13/T20/T21 与 docs/22/24/25 的分散结论落成一张可施工矩阵。
> 身份模型：**未登录 / 普通用户（租户成员）/ 管理员** 三级；规模 ≤5 人（T1），管理员=实例所有者。
> 判断原则：① 凡是"花了实例所有者的钱或密钥"的 → 管理员托管；② 凡是"个人投资行为数据" → 租户私有；③ 凡是"市场公共计算" → 全局共享无人称；④ 凡是"能搞坏整个实例"的 → 管理员专属。

## 1. 三级身份定义

| 身份 | 定义 | 来源 |
|---|---|---|
| 未登录 | 无任何凭证 | 仅可访问 /login、/setup（仅 users 空表时可用）、/reports 签名链接（MT-P2 起） |
| 普通用户 | users.role=user，属于某 tenant | 管理员邀请创建（T12），MT-P1 上线即关注册 |
| 管理员 | users.role=admin，默认 tenant_id=1 | env AUTH_USERNAME/PASSWORD 或 app_settings auth_* 迁移而来（T20） |

## 3. 关键经验判断（用户授权委托项，逐条说明）

1. **数据源管理归管理员**：普通用户改数据源优先级会影响全实例所有租户的取数链路，属"能搞坏整个实例"类。
2. **策略标定/回测触发归管理员**：strategy_weights/factor_weights 是全局自动标定（T19），触发回测消耗公共算力；普通用户只读信号结果。
3. **普通用户可见数据源/Agent 运行状态（只读）**：便于自查"为什么没推送"，但不给写权限。
4. **AI 服务两态 UI**：共享配额租户在 AI 服务页只看到托管服务下拉（无密钥输入框）；非共享租户才有完整自建表单——这个分化在 MT-P4 前端落地，MT-P1 先在 API 层按 quota_shared_with_admin 判定。
5. **管理员也是租户成员**：admin 属于默认租户（tenant_id=1），日常盯盘功能与普通用户同一套界面，只是多出管理菜单——不搞独立管理后台（≤5 人规模不值得）。
6. **MT-P1 行为等价单用户**：上线即只有 admin 一个用户（T12），所有现有功能路径不变；普通用户路径在 MT-P2 数据隔离落地后才有意义。

## 4. MT-P1 落地清单（本矩阵的施工映射）

- users/tenants 表（docs/21 DDL，create_all 新表惯例；v120 时自包含 DDL 兜底）
- auth.py 重写：JWT claims（sub/tenant_id/role/pwd_at）、bcrypt+透明重哈希、/auth/me、邀请制建用户（admin-only）、/setup bootstrap、废除未设密码放行、旧 token 一次性踢出（缺 claim 即 401）、改密校验旧密码
- 机制点落地：TenantContext contextvar + 中间件 set/reset + do_orm_execute（MT-P1 单租户直通模式，MT-P2 激活过滤）+ default-deny 表注册表骨架 + tenant_scope()
- settings.py PUT key 白名单 + 实例级 key 管理员限定；logs.py DELETE 管理员限定
- 前端：登录态真实身份、AccountMenu 显示用户名/角色、管理员用户管理页（邀请/列表/配额开关）、localStorage 账号切换检测清空
- 测试：conftest authenticated_client fixture、JWT/bcrypt/邀请制/白名单新用例、611 基线零修改全绿
