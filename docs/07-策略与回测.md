# 07 · 策略 / 因子 / 回测 / 信号

> 范围：`src/core/strategy_engine.py`、`strategy_catalog.py`、`factor_eval.py`、`factor_weights.py`、`signals/`、`backtest/`、`quant_adapters.py`。
> 面向二次开发者的结论性文档；所有结论均来自源码阅读，不确定处已标注。

## 1. 模块职责一句话

把 EntryCandidate（候选池，由 `entry_candidates.py` 产出，范围外）按「策略目录 × 因子打分 × 市场环境 × 组合约束」合成每日策略信号（`StrategySignalRun`），事后用真实 K 线评估前向收益（`StrategyOutcome`），再反过来驱动策略权重与因子权重的自动校准，构成一个「信号 → 后验 → 调权」的闭环；另带一个纯 Python、零依赖的事件式回测内核（`backtest/`）作为验证与模拟盘成本模型的地基。

## 3. 核心机制与数据流

### 3.1 信号生成主流程 `refresh_strategy_signals()`

```
refresh_entry_candidates（可选 rebuild，范围外）
  → 读当日 EntryCandidate（按 score 排序，≤2000 条）
  → _upsert_market_regime_snapshots：按市场算 regime（bullish/neutral/bearish，
    score = 0.45·上涨宽度 + 0.30·平均涨幅 + 0.25·active 比例）
  → _build_cross_section_features：市场内截面排名百分位（score/涨幅/换手/量比）
    → relative_strength_pct、crowding_risk（拥挤度惩罚）
  → _load_news_metrics：NewsCache 近 72h，关键词正/负事件偏置 + 时效衰减 → event_score/event_bias
  → 每候选 × 每策略代码（来自 strategy_tags / candidate_source 映射）：
      _compute_factor_breakdown 打分 → upsert StrategySignalRun
  → _apply_portfolio_constraints：组合层降级（demote → watch + 分数封顶）
  → 清理同日 stale 行 → _sync_factor_and_risk_snapshots 写因子/风险快照
```

### 3.2 因子打分 `_compute_factor_breakdown()`（核心，二开必看）

六个因子 + 来源加成，全部有明确 clamp 边界：

- **alpha_score**：`(base_score-50)*0.45`（[-12,18]），可加相对强度修正
- **catalyst_score**：涨幅区间（1–7% 加 4 分）、信号文本关键词（突破/回踩/超跌）、event_score×0.55、相对强度
- **quality_score**：`(plan_quality-50)/5`（[-8,10]）
- **risk_penalty**（惩罚项，合成时减）：高风险策略 +1.5、|涨幅|≥8% +2、非 active +2.5、plan_quality<70 +1.5、负事件 +2.2
- **crowd_penalty**（惩罚项）：涨幅≥9%、量比≥3、成交额≥80 亿、crowding_risk
- **source_bonus**：v1 权重固定 1.0，不进标定集

合成公式（strategy_engine.py:881-896）：

```python
raw_score = base_score + fw·alpha + fw·catalyst + fw·quality + source_bonus
raw_score -= fw·risk_penalty + fw·crowd_penalty
final_score = clamp(raw_score × 策略权重 × regime_multiplier, 0, 100)
```

- `fw` 来自 `factor_weights.get_factor_weights(market)`（默认 1.0 = 零回归基线）
- **快照存 raw 因子分、权重只作用于合成**——这是 IC 测量不失真的关键设计（见 factor_calibration.py 头注释），二开时不要改成"快照存加权后分数"
- regime_multiplier：bullish ×1.06/1.01，bearish ×0.90/0.97，clamp [0.85, 1.12]
- 语义对齐硬编码封顶：无入场区间的 buy/add ≤ 66 分、hold/watch ≤ 78（持仓）/65（未持仓）、非 active ≤ 69

### 3.3 组合约束 `_apply_portfolio_constraints()`

按市场三道闸门，超限信号被 `_demote_signal()`（status=inactive、action=watch、分数封顶、payload 记 `constraint_reasons`）：

1. 未持仓 active 机会数上限：CN 30 / HK 20 / US 20
2. 高风险占比上限：CN 35% / HK 32% / US 30%
3. 单一策略占比上限 42%（`MAX_SINGLE_STRATEGY_SHARE`）

### 3.4 后验评估 `evaluate_strategy_outcomes()`

- 输入：近 60 天、action ∈ buy/add/hold/watch 的 `StrategySignalRun`（上限 1200 条）
- 口径：horizon ∈ {1,3,5,10} 自然日；base = entry_low/high 中值（缺省回退 source_meta 报价 → 信号日收盘）；outcome = target_day 当日或之前最近收盘；return% = (outcome-base)/base×100
- 命中判定：outcome ≥ target_price → hit_target；≤ stop_loss → hit_stop；否则 evaluated
- 防泄漏：target_day > today 则跳过；**用自然日而非交易日**（与 backtest 的交易日口径不同，注意）
- 幂等：`(signal_run_id, horizon)` 已评估则跳过；每 50 条分批 commit（避免与 60s 调度器抢 SQLite 写锁）

### 3.5 权重校准双闭环

**策略级** `rebalance_strategy_weights()`：近 45 天 StrategyOutcome 聚合胜率/均收益 → `target = default × (1 + 0.45·win_term + 0.35·ret_term)`，clamp [0.45, 1.90]，EMA（alpha=0.35）平滑，min_samples=8，写 `StrategyWeight` + `StrategyWeightHistory`。

**因子级** `calibrate_all_markets()`（factor_calibration.py）：`evaluate_factor_ic()` 算每因子 IC/IR → `compute_target`（优先 IR/0.5，fallback IC/0.05，惩罚因子翻符号，`target = 1 + 0.4·term`）→ EMA + clamp [0.5, 1.5] → 写 `FactorWeight`/`FactorWeightHistory`。门控：`is_pinned`、`auto_calibrate=False`、样本 < 20、IC 缺失均跳过。

### 3.6 回测内核 `backtest/`

- 入场：信号日**下一交易日开盘价**（`first_index_after` 严格大于，无未来函数）；T+1 起才可卖
- 逐日检查：同日双触保守判先止损；开盘跳空越过止损/止盈按开盘价成交；最大持有 `holding_days`（默认 10）按收盘平
- 仓位：默认每笔固定 10 万元、100 股整数倍（`fixed_cash_sizer`，可注入自定义 sizer）
- 净值曲线：按平仓日累加已实现盈亏（**不含浮动持仓逐日 mark**，TODO）
- 已知未建模：涨跌停无法成交约束
- `horizon_return()` 复刻 `evaluate_strategy_outcomes` 口径供交叉验证
- **现状：回测内核未被任何 API 路由或 Agent 直接调用**，仅 `paper_trading_engine.py` 复用其 `CostModel`；`quant_adapters.available_backends()` 暴露于 `GET /api/paper-trading/backends`

### 3.7 SignalPack（Agent 输入打包）

`SignalPackBuilder.build_for_symbols()` 是 5 个 Agent（premarket_outlook / intraday_monitor / chart_analyst / news_digest / daily_report）的统一数据源：
- 单实例内存缓存（quote/kline/news/flow/events 五个 dict），**仅单次运行有效**，跨运行不复用
- 数据源策略 `_source_policy`：DB `DataSource` 表无该类型 → 用默认 provider；有但全禁用 → disabled；有启用 → 按 priority 尝试
- 实际只支持：quote/kline = tencent，capital_flow/events = eastmoney（其他 provider 记日志跳过）
- 资金流仅 CN；news/events 各截取前 5 条入包；`missing` 字段显式记录缺哪类数据

## 4. 状态存储位置

全部为 SQLite DB 表（`src/web/models.py`），无 JSON 文件状态：

| 表 | 内容 |
|---|---|
| `StrategyCatalog` | 策略目录（7 内置策略 seed，name/risk_level/default_weight/params.horizon_days） |
| `StrategySignalRun` | 每日信号主表（快照日 × 候选 × 策略；payload 含 score_breakdown/market_regime/cross_feature/news_metric/constrained） |
| `StrategyFactorSnapshot` | 每信号的 raw 因子分（signal_run_id 关联），IC 测量的输入 |
| `StrategyOutcome` | 前向收益（signal_run_id × horizon 唯一） |
| `StrategyWeight` / `StrategyWeightHistory` | 策略权重（regime + market 维度）及审计 |
| `FactorWeight` / `FactorWeightHistory` | 因子权重（5 因子 × CN/HK/US），meta 存 last_ic/last_ir |
| `MarketRegimeSnapshot` / `PortfolioRiskSnapshot` | 每日 × 市场的 regime 与组合风险快照 |
| `EntryCandidate` / `NewsCache` / `DataSource` | 输入侧（范围外文件维护） |

## 5. 对外接口

**API 路由**（均需鉴权）：
- `GET /api/recommendations/strategy-catalog`、`/strategy-signals`、`/strategy-regimes`、`/strategy-risk-snapshots`、`/strategy-factors/{signal_run_id}`、`/strategy-stats`、`/strategy-factor-ic`、`/strategy-weight-history`
- `POST /api/recommendations/strategy-signals/refresh`、`/strategy-signals/outcomes/evaluate`、`/strategy-weights/rebalance`
- `GET/POST /api/factors/weights[...]`（因子权重只读 + 手动覆盖/pin/开关自动标定）
- `GET /api/paper-trading/backends`（回测后端探测）

**调度器**（`context_scheduler.py`，UTC 时区）：
- 机会刷新：cron 01:15 / 05:30 / 14:00（= 北京 09:15 / 13:30 / 22:00），`refresh_strategy_signals(rebuild_candidates=True)`
- 后验评估：interval（`eval_interval_hours`）+ 启动后 15s 补跑；顺序为 evaluate_outcomes → rebalance → calibrate（**因子标定必须在 outcome 评估之后**，不能并行）
- 所有 job 带 jitter=120s、max_instances=1，防与 60s 级调度器同刻写 SQLite

## 6. 二次开发扩展点与注意事项

**扩展点**
- 加策略：在 `DEFAULT_STRATEGIES` 加 `StrategySpec`（`ensure_strategy_catalog` 自动 seed/同步 name 等字段，但不会覆盖已有 `enabled` 和用户改过的 `params`）；候选侧需让 `EntryCandidate.strategy_tags` 含该 code，否则映射回退 `watchlist_agent`
- 加因子：需同步改 4 处——`factor_weights.CALIBRATABLE_FACTORS`、`factor_eval.FACTOR_FIELDS`、`_compute_factor_breakdown` 打分逻辑、`StrategyFactorSnapshot` 模型列；否则标定闭环与新因子脱节
- 换回测后端：实现 `quant_adapters.BacktestAdapter` 协议（`name` + `run(signals, bars_by_symbol)`）；选型参考 `.docs/quant-framework-comparison.md`
- 自定义仓位：给 `Backtester(sizer=...)` 注入 `price -> qty` 函数

**坑与注意事项**
1. **两套收益口径不一致**：`evaluate_strategy_outcomes` 用自然日 horizon + 信号日基准价；`backtest` 用交易日 + 次日开盘入场。对比结果时用 `horizon_return()` 交叉验证，别直接比数字。
2. 因子权重改动只影响**新合成**的 final_score；历史 `StrategyFactorSnapshot.final_score` 不会回刷——做 A/B 对比时注意。
3. `_apply_portfolio_constraints` 的 demote 直接改 DB 行并封顶分数，API 层 `_normalize_action_view` 又做一次封顶；两处封顶值（66/69/78/65 等）分散在多处，改阈值需全局搜索。
4. `_load_news_metrics` 的股票名回退匹配是**子串匹配**（`name in text`），短名股票（如"平安"）可能误挂新闻。
5. `get_factor_weights` 有写副作用（lazy seed 插入缺失行），在只读上下文调用也会产生 commit；`set_factor_weight` 手动权重 clamp 到 [0.1, 3.0]，自动标定 clamp 到 [0.5, 1.5]，两套边界不同。
6. `SignalPackBuilder` 的 provider 支持面窄：配了非 tencent/eastmoney 的数据源会被静默跳过（仅 info 日志），quote 拿不到时 missing=["quote"] 但不报错。
7. 回测净值曲线不含浮动盈亏、不建模涨跌停，结果偏乐观；用于对外展示前需声明口径。
8. 调度任务全部直连 `SessionLocal()` 写 SQLite，长事务已做分批 commit（50 条）；二开新增写路径时保持这个模式，避免与 price_alert/paper_trading 的 60s 任务死锁。
9. `structured_output.py` 的 action 白名单（buy/add/reduce/sell/hold/watch/alert/avoid）与前端展示联动，新增 action 需同步前端。
10. `signal_pack.py` 中 `NewsCollector.from_database()` / `EventsCollector.from_database()` 读取 DB 配置，事件源禁用时不抛错而是 `missing` 标记——Agent 侧要消费 `missing`/`sources` 字段而非假设数据齐全。
