## 1. 四方共识的高危问题（必须修订）

### C1. T7「事件门市场级共享」语义不自洽，需状态分层（架构+测试一致，高）
- 代码事实：`intraday_event_gate.py:309-331` 的 playbook 价位穿越冷却键是 `方向:位名@价位`，**不含租户**——两租户对同一标的设不同 playbook，冷却互吞（1800s 内后触发者被吞）；`_load_playbook_levels`（:206-227）按 symbol 无过滤读取，会取到别租户的档案价位。
- 但行情类状态（last_price 基线、tech_sig）确实是市场级，按租户复制反而错误。
- **修订：状态分层**——市场观测态（last_price/tech_sig）单份共享；租户态（pb_fired 冷却、方案价位解析）按 `(tenant, symbol)` 键控；保留 per-tenant 执行模型。T7 原表述"状态文件保持单份"过于粗糙。

### C2. stock_context_snapshots 误判为共享红利，必须改判私有（研发，泄漏级）
- 代码事实：`context_builder.py:487/539/556` 把**持仓约束 + playbook 摘要写进 payload 落库**，且快照记忆会**回读注入下次 prompt**。若照 v1.0 §1.3 作为共享红利保留，租户 A 的持仓数量和方案价位会注入租户 B 的 LLM 上下文——这是内建进方案的泄漏。
- `news_topic_snapshots` 同理（输入是租户 watchlist 的新闻）。
- **修订：两表从红利清单删除，判为强私有**；§1.3 红利清单只保留 market_regime/market_scan/news_cache/marketdata 缓存。

### C3. 现有备份机制不可用于迁移兜底，v120 前置任务=先修备份（运维，准致命遗漏）
- 代码事实：WAL 模式（`database.py:31`）+ `_backup_db_before_migration` 用 `copy2` **只拷主文件不拷 -wal/-shm**（:114-131），无 checkpoint——备份不是一致性快照且可能丢最近事务；无轮转、与主库同卷。
- **修订：v120 前置任务**——备份改为 `wal_checkpoint(TRUNCATE)` 后拷贝或 `VACUUM INTO`；恢复后 `integrity_check`；轮转 ≤3 份；出卷冷备。这条不修，T6 的"迁移失败回滚"承诺是空头支票。

### C4. Watchtower 回滚死结（运维，高）
- 回滚旧镜像后 5 分钟内 Watchtower 又把 latest 拉回来，CLEANUP=true 还会删旧镜像。v1.0 只登记了风险无处置。
- **修订（T11 发布门禁）**：含破坏式迁移的版本**永不直接推 latest**（独立 tag + 人工 promote）；发布 SOP：暂停 Watchtower → 出卷冷备 → 升级 → 验证 → 恢复。可选 MIGRATION_ACK 环境门禁。

### C5. tenant_id 列必须 NOT NULL DEFAULT 默认租户——决定 611 测试生死（测试，高）
- 代码事实：`conftest.py:75-87` 只 `create_all`，大量用例直接构造 ORM 行。tenant_id 若无默认值，611 基线会大面积 IntegrityError。
- **修订（T18）**：所有私有表 tenant_id = `NOT NULL DEFAULT 1`（默认租户），611 基线零修改全绿（仅允许改 conftest）；FK 链表可免逐行回填。子行 tenant 只允许从父实体派生（架构不变量，对账 SQL 进验收）。

### C6. stocks 表无 symbol 唯一约束，12 处解析点多租户随机命中（研发，方案最大遗漏）
- 代码事实：全仓 **12 处** `Stock.symbol==X .first()` 解析点（insights.py:177,322、morning_brief.py:64、daily_report.py:274、context_builder.py:438、tradingagents/backfill.py:90、data_collector.py:172、intraday_event_gate.py:216、stocks.py:223,387、chat.py:354、templates.py:213）——多租户下同代码命中哪行不确定。
- **修订（T15）**：stocks 每租户复制 + 12 处改租户作用域 + 新增 `(tenant_id, symbol, market)` UQ。

### C7. 身份传播必须用机制而非人肉（架构+研发一致，高）
- 全仓 455 处查询点靠逐条人工审计，"漏一条=泄漏"不可防守；`log_handler.py`、`notify_dedupe.py:54`、`context_store.py` 等**不在任何调用链上**，参数透传覆盖不了。
- **修订（T16）**：web request 层用 **contextvar TenantContext + SQLAlchemy `do_orm_execute` 事件自动加 tenant 过滤 + default-deny 查询 helper**——把 N 个人工点降为 1 个机制点；后台调度链显式传参（调度 job 命名 + `context_builder(tenant_id, agent_name)` 签名**前置重构**，不可打补丁）。

### C8. 调度扇出必须单 job 遍历租户（研发+运维一致，高）
- per-tenant 注册 N 个 job 是"真正的炸弹"：写次数 ∝ 触发数而非租户数，N 个 job 会把调度器和上游行情接口打爆（market_http.py 无全局限流器）。
- **修订（T17）**：单 job 内遍历租户；NotifyThrottle 不必重建 UQ，tenant 编进 scope 字符串（`__notify__:{tenant}:{hash}`）。

## 4. 四方各自的独立贡献（非共识但已采纳）

- **架构师**：tenant_id 一致性不变量——子行 tenant 只允许父实体派生，对账 SQL 进验收；agent override 语义=整体替换；优先级链 `stock_agent > tenant_override > template > 系统默认`；邀请制注册；配置包导出管理员专属。
- **研发**：迁移每步幂等可重入 + 清 `__new` 半成品表；tenants/users 建表并入 v120 自包含；SQLite ADD COLUMN 不能同时 NOT NULL DEFAULT+FK；事务内 PRAGMA foreign_keys 是 no-op。
- **测试QA**：§8 分期门禁整体升级为"自动化必过 + 人工签字"双栏（G1–G5）；迁移验收 4 制品（M7）；prompt 内容级跨租户断言（M8）；禁止更新 prompt 快照哈希。
- **运维**：pre-flight 磁盘 ≥2×DB；RTO ≤30min；迁移窗口=服务全停（init_db 在监听前）；N≤20 硬上限写进 T1；备份脱敏纳入最小集。
