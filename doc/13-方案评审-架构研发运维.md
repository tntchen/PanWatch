> 结论：三方一致判定方案**方向正确、可执行，但有 3 项高危误判必须修订**；已全部回填计划 v1.3

## 1. 三方共识的高危问题（必须修订）

### C1. 龙虎榜数据源已存在，P3b「新增 collector + vendor 注册」是重复建设（三方一致，高）
- 代码事实：`packages/marketdata/.../registry.py:82` 已注册 `EastmoneyDragonTigerVendor`；`client.py:292` 有 `dragon_tiger(date=...)` 门面（市场级、300s TTL）；`server.py:567-579` 的 `DATA_SOURCE_SEEDS` 已含 dragon_tiger seed。
- 修订：P3b 降级为「新增 host 侧 accessor `md_dragon_tiger()` + daily_report 消费 + 按个股过滤」，**不碰 DATA_SOURCE_SEEDS**。决策点 D8 按错误前提确认，已回填修订。
- 运维补充：东财接口 18:00 是否出齐需**实测验证**，这是 daily_report 改 18:00 的前置项；该 type 无备源，daily_report 必须容错跳过龙虎榜章节而非整体报错。

### C2. 触发器扩展方案修正：不需要跨日状态，也不需要新调度器（三方一致，高）
- `turnover_rate`（换手率）**已在 quote dict 里**（`marketdata_client.py:81`），引擎侧约 3 行快赢——但只有 tencent vendor 填充，港美股无此字段，必须 **CN-only 双层门控**（表单 + 引擎）。
- `consecutive_close`（连续 N 日收盘）：`KlineCollector.get_kline_summary()` 已从日 K 本地计算，引擎对 volume_ratio 已有 K 线回退先例（`price_alert_engine.py:166-172`）——**直接取最近 N 根日 K 无状态评估**，删掉"状态存规则 JSON"（运维指出：condition_group 是用户可编辑配置，混存状态会被编辑回传覆盖/触发 400）。
- `capital_flow`（主力净流入）：仅 A 股；接入走 `CapitalFlowCollector` 既有 **600s TTL 缓存**，避免 N 条规则 × 60s 直拉接口的反爬风险。
- 架构师结论：路线 A 修正为「混合 A」——即时值条件进现有引擎，日级条件无状态 K 线评估，复用 PriceAlertHit 与频控/通知，不新建调度器、不重复造轮子。

### C3. position_trades 的关仓与台账一致性语义必须预先定义（架构+研发，高）
- `Position` 无 status 列，Account→Position 是 `cascade="all, delete-orphan"`：关仓若删行，流水被级联清空。
- 修订为三条规定：① 关仓 = quantity=0 **保留行**；② 有流水的持仓**禁止删除**；③ 手动编辑成本/数量降级为自动生成 **adjustment 流水**（杜绝双写）；④ 重新买入已关仓股票走 trades 端点而非 POST /positions（现有 400 拒绝逻辑不变）。
- 精度：计算用 `Decimal`（落库 float），断言容差写进测试——否则验收用例"3150 股 @112.572"末位漂移必翻车。
- `invested_amount` 减仓时按比例结转（口径写明）。

## 4. 三方各自的独立贡献（非共识但已采纳）

- **架构师**：fork 冲突面原则——新文件优先于改旧文件；playbook 表结构定为 `(stock_id, version, is_active, payload JSON)` + payload 内含 `schema_version` 信封，演进只增不改；`stock_agents.schedule` 废弃字段要么修要么 UI 标注，不留着。
- **研发**：测试策略对齐 tests/ 惯例（conftest mock notifier、清 collector 缓存）——P1 纯函数成本重算单测 + API sqlite 测试；P2 参照 `test_price_alert_volume_ratio.py` 模式；P3 prompt 组装断言（有档案出现摘要、无档案 prompt 不变）+ `test_portfolio_*` 回归网；新简报 Agent 的建议词表要与 `*_ACTION_MAP` 同步设计。
- **运维**：可观测性——两个新简报 Agent 走 AgentScheduler 自动被 selfcheck 覆盖，无需新增注册；新增调度任务确认 `max_instances=1` + 错峰分钟；单日新增推送 <20 条，现有频控兜得住。
